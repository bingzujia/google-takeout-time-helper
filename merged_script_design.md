# 统一照片时间戳修复脚本 — 详细设计文档

## 一、脚本定位与核心目标

### 1.1 功能定义

将 `fix_img_timestamps.sh`（文件名时间戳提取）与 `fix_takeout_photo_time_wsl.sh`（JSON 元数据时间戳提取）合并为单一脚本，实现：

- **所有文件**：时间戳优先从文件名提取（支持 12 种格式，见 2.1 和 2.4），GPS/设备信息从 JSON 补充
- **无 JSON 配对的文件**：仍可处理（仅时间戳，无 GPS/设备信息）
- **文件名无法解析时间的文件**：回退到 JSON `.photoTakenTime.timestamp`，若 JSON 也无时间戳则跳过

### 1.2 运行环境要求

| 依赖项 | 用途 | 安装命令 |
|--------|------|----------|
| `bash` (≥4.0) | 脚本解释器 | 系统预装 |
| `touch` (支持 `-d` 参数) | 修改文件时间戳 | 系统预装 |
| `jq` | 解析 JSON 元数据 | `sudo apt install jq` |
| `find` | 文件递归查找 | 系统预装 |
| `stat` (支持 `-c %Y`) | 读取文件当前修改时间 | 系统预装 |
| `date` (支持 `-d` 参数) | 时间格式转换与验证 | 系统预装 |
| `file` | 检测文件类型（错误诊断用） | 系统预装 |
| `df` (支持 `-T` 参数) | 检测文件系统类型（错误诊断用） | 系统预装 |

### 1.3 支持的运行环境

| 环境类型 | 检测方式 | 支持状态 |
|----------|----------|----------|
| WSL (Windows Subsystem for Linux) | `/proc/version` 包含 `microsoft`（不区分大小写） | 完全支持 |
| Linux 原生系统 | `/proc/version` 包含 `linux`（不区分大小写） | 完全支持 |
| macOS | 无 `/proc/version` 文件 | 不支持（`touch -d` 语法不同） |
| Windows CMD/PowerShell | 无 bash 环境 | 不支持 |

## 二、文件分类体系

### 2.1 文件名时间戳格式总表（所有文件统一适用）

**所有文件**（包括 IMG/VID 和非 IMG/VID）在时间戳提取时，均按以下 12 种格式逐一尝试（按匹配优先级排列）：

| 序号 | 正则表达式模式 | 示例文件名 | 提取的时间 | 特殊处理 | 来源 |
|------|----------------|------------|------------|----------|------|
| 1 | `^(IMG\|VID)(\d{4})(\d{2})(\d{2})(\d{2})(\d{2})(\d{2})` | `IMG20250409084814.MOV` | `2025-04-09 08:48:14` | 锚定文件名开头，仅匹配 IMG/VID 前缀 | IMG/VID 专用 |
| 2 | `^(IMG\|VID)_(\d{4})(\d{2})(\d{2})_(\d{2})(\d{2})(\d{2})` | `IMG_20250727_141938.MOV` | `2025-07-27 14:19:38` | 锚定文件名开头，仅匹配 IMG/VID 前缀 | IMG/VID 专用 |
| 3 | `(\d{8})_(\d{6})` | `IMG_20230302_112040.jpg` | `2023-03-02 11:20:40` | 前8位为日期，后6位为时间 | 通用 |
| 4 | `(\d{8})(\d{6})` | `IMG20230123102606.jpg` | `2023-01-23 10:26:06` | 连续14位数字 | 通用 |
| 5 | `(\d{8})_(\d{3,6})` | `WP_20131010_074.jpg` | `2013-10-10 12:00:00` | 时间不足6位时，默认使用中午12点（`074` → 时间部分缺失，设为 `12:00:00`） | 通用 |
| 6 | `(\d{8})_(\d{6})~\d+` | `20151120_120004~2.jpg` | `2015-11-20 12:00:04` | 忽略 `~` 后的连拍序号 | 通用 |
| 7 | `(\d{4})-(\d{2})-(\d{2})-(\d{2})-(\d{2})-(\d{2})` | `Screenshot_2016-02-28-13-06-34.png` | `2016-02-28 13:06:34` | 6组数字用连字符分隔 | 通用 |
| 8 | `(\d{8})-(\d{6})` | `Screenshot_20210803-084525.png` | `2021-08-03 08:45:25` | 日期时间用单个连字符分隔 | 通用 |
| 9 | `mmexport(\d{13})` | `mmexport1491013330299.jpg` | 取前10位毫秒时间戳 → 秒级 | 13位数字为毫秒级 Unix 时间戳 | 通用 |
| 10 | `mmexport(\d{13})-([[:alpha:]]+\|[[:punct:]]+\|[[:digit:]]+\|[一-龥]+)` | `mmexport1491013330299-已修改.jpg` | 同上 | 带中文/英文/符号后缀 | 通用 |
| 11 | `mmexport(\d{13})\(\d+\)` | `mmexport1491013330299(1).jpg` | 同上 | 带括号编号后缀 | 通用 |
| 12 | 兜底（无匹配） | — | — | 进入 JSON 回退 | — |

**匹配优先级说明**：
- 格式1-2 优先匹配：因为使用 `^` 锚定文件名开头，且限定 `IMG`/`VID` 前缀，精确度最高
- 格式3-11 次之：通用模式，不锚定开头，在格式1-2 失败后逐一尝试
- 格式12 为兜底：所有格式均失败时，进入 JSON 回退

**支持的扩展名**（不区分大小写）：

| 类别 | 扩展名列表 |
|------|------------|
| 图像 | `jpg`, `jpeg`, `png`, `gif`, `bmp`, `tiff` |
| 视频 | `mov`, `mp4`, `avi`, `mkv`, `wmv`, `flv`, `webm`, `m4v`, `3gp` |

### 2.2 非 IMG/VID 文件定义

**判定条件**：不符合 2.1 中 IMG/VID 文件名模式的所有文件。

**常见类型及时间戳来源**：

| 文件类型 | 文件名示例 | 时间戳来源 | 说明 |
|----------|------------|------------|------|
| 截图 | `Screenshot_2023-05-01-22-08-49-61.png` | 文件名优先（2.4 格式5/6），JSON 回退 | 文件名格式见 2.4 |
| 微信导出 | `mmexport1491013330299.jpg` | 文件名优先（2.4 格式7-9），JSON 回退 | 文件名格式见 2.4 |
| Windows Phone | `WP_20131010_074.jpg` | 文件名优先（2.1 格式5），JSON 回退 | 文件名格式见 2.1，时间不足6位时默认 12:00:00 |
| 纯数字日期 | `20151120_120004.jpg` | 文件名优先（2.4 格式1/2/4），JSON 回退 | 文件名格式见 2.4 |
| 其他任意文件 | `任意文件名.jpg` | 必须依赖 JSON | 文件名无法解析时间时使用 |

### 2.3 JSON 元数据文件定义

**判定条件**：扩展名为 `.json` 的文件。

**支持的 JSON 文件名格式**（按匹配优先级排列）：

| 序号 | 正则表达式模式 | 示例 JSON 文件名 | 解析出的 base_name | 解析出的 number | 说明 |
|------|----------------|------------------|-------------------|-----------------|------|
| 1 | `^(.+)\.([^.]+)\.([^.]+)\((\d+)\)\.json$` | `IMG_20240913_162956.jpg.supplemental-metadata(1).json` | `IMG_20240913_162956(1).jpg` | `1` | 带编号 + 带补充后缀 |
| 2 | `^(.+)\((\d+)\)\.json$` | `IMG_20240913_162956(1).json` | `IMG_20240913_162956(1)` | `1` | 带编号 |
| 3 | `^(.+)\.([^.]+)\.([^.]+)\.json$` | `IMG_20240913_162956.jpg.supplemental-metadata.json` | `IMG_20240913_162956.jpg` | 无 | 带补充后缀，无编号 |
| 4 | `^(.+)\.\.json$` | `IMG_20240913_162956..json` | `IMG_20240913_162956` | 无 | 双点号格式 |
| 5 | `^(.+)\.json$` | `IMG_20240913_162956.json` | `IMG_20240913_162956` | 无 | 标准格式 |
| 6 | 兜底（不匹配以上任何模式） | `任意名称.json` | 去掉 `.json` 后的部分 | 无 | 直接截断 |

## 三、JSON 元数据结构

### 3.1 时间戳字段

| JSON 路径 | 数据类型 | 示例值 | 说明 |
|-----------|----------|--------|------|
| `.photoTakenTime.timestamp` | 字符串（Unix 时间戳，秒级） | `"1683012040"` | 拍摄时间，**主要时间来源** |
| `.photoTakenTime.formatted` | 字符串（人类可读格式） | `"May 2, 2023, 10:20:40 AM UTC"` | 仅用于调试，不用于时间设置 |

### 3.2 GPS 信息字段

| JSON 路径 | 数据类型 | 示例值 | 说明 |
|-----------|----------|--------|------|
| `.geoData.latitude` | 数字 | `39.9042` | 纬度（十进制度） |
| `.geoData.longitude` | 数字 | `116.4074` | 经度（十进制度） |
| `.geoData.altitude` | 数字 | `50.0` | 海拔（米） |
| `.geoData.latitudeSpan` | 数字 | `0.0` | 纬度精度范围 |
| `.geoData.longitudeSpan` | 数字 | `0.0` | 经度精度范围 |

### 3.3 设备信息字段

| JSON 路径 | 数据类型 | 示例值 | 说明 |
|-----------|----------|--------|------|
| `.cameraMake` | 字符串 | `"Apple"` | 设备制造商 |
| `.cameraModel` | 字符串 | `"iPhone 14 Pro"` | 设备型号 |

### 3.4 其他元数据字段

| JSON 路径 | 数据类型 | 示例值 | 说明 |
|-----------|----------|--------|------|
| `.title` | 字符串 | `"IMG_20240913_162956.jpg"` | 原始文件名 |
| `.description` | 字符串 | `""` | 描述（通常为空） |
| `.url` | 字符串 | 内部 Google 链接 | Google Photos 内部链接，无外部访问价值 |
| `.imageViews` | 字符串 | `"0"` | 浏览次数 |
| `.favorited` | 布尔值 | `false` | 是否收藏 |
| `.archived` | 布尔值 | `false` | 是否归档 |

## 四、处理逻辑流程图

### 4.1 整体执行流程

```
┌─────────────────────────────────────────────────────────────┐
│ 阶段1: 环境初始化                                             │
│  ├─ 解析命令行参数 (--dry-run)                               │
│  ├─ 检测运行环境 (WSL / Linux)                               │
│  ├─ 检查依赖工具 (touch, jq, find)                           │
│  └─ 测试 touch -d 参数可用性                                 │
├─────────────────────────────────────────────────────────────┤
│ 阶段2: 文件发现与分类                                         │
│  ├─ 查找所有 "Photos from*" 目录                             │
│  ├─ 统计目录内所有 .json 文件数量                             │
│  ├─ 统计所有 IMG/VID 文件数量（递归，不限目录）                │
│  └─ 若 JSON 数量 = 0 且 IMG/VID 数量 = 0 → 退出              │
├─────────────────────────────────────────────────────────────┤
│ 阶段3: 构建处理队列                                           │
│  ├─ 遍历每个 "Photos from*" 目录                             │
│  │   └─ 对每个 .json 文件:                                   │
│  │       ├─ 解析 JSON 文件名 → 得到 base_name + number       │
│  │       ├─ 查找对应照片文件（精确匹配 → 修饰符匹配 → 模糊匹配）│
│  │       ├─ 若找到照片 → 加入处理队列                        │
│  │       └─ 若未找到 → 记录跳过                              │
│  └─ 遍历所有 IMG/VID 文件（递归）                            │
│      └─ 若该文件未被 JSON 队列覆盖 → 加入独立处理队列         │
├─────────────────────────────────────────────────────────────┤
│ 阶段4: 逐文件处理                                            │
│  └─ 对队列中每个文件执行 4.2 的时间戳决策树                   │
├─────────────────────────────────────────────────────────────┤
│ 阶段5: 结果统计与输出                                         │
│  └─ 输出: 总计 / 成功 / 跳过 / 失败                          │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 时间戳决策树（核心逻辑）

对每个待处理的照片文件，按以下顺序决策：

```
照片文件
│
├─ 步骤1: 从文件名提取时间戳（所有文件统一处理）
│   │
│   ├─ 尝试 12 种文件名格式（2.1 中的12种，按优先级逐一匹配）
│   │   ├─ 格式1-2 匹配成功（IMG/VID 专用） → 验证日期有效性 → 有效 → 使用此时间戳 → 跳至步骤3
│   │   ├─ 格式3-11 匹配成功（通用） → 验证日期有效性 → 有效 → 使用此时间戳 → 跳至步骤3
│   │   └─ 全部失败 → 进入步骤2（JSON 回退）
│
├─ 步骤2: JSON 回退（仅当文件名无法解析时间时触发）
│   ├─ 存在配对的 .json 文件？
│   │   ├─ 是 → 读取 .photoTakenTime.timestamp
│   │   │   ├─ 有值 → 使用此时间戳 → 跳至步骤3
│   │   │   └─ 无值/null → 跳过该文件，记录原因"无可用时间戳"
│   │   └─ 否 → 跳过该文件，记录原因"无可用时间戳"
│
└─ 时间戳应用
    ├─ 获取文件当前 mtime (stat -c %Y)
    ├─ 计算与目标时间戳的差值
    ├─ 差值在 [-1, +1] 秒范围内？
    │   ├─ 是 → 跳过（时间戳已正确）
    │   └─ 否 → 执行 touch -d
    │       ├─ 成功 → 计数 +1
    │       └─ 失败 → 记录错误详情（权限/文件类型/文件系统）
```

### 4.3 照片文件匹配流程（JSON → 照片）

当处理 `.json` 文件时，需要找到对应的照片文件：

```
输入: JSON 文件路径 (如 Photos from 2024/IMG_20240913_162956.jpg.supplemental-metadata(1).json)
│
├─ 步骤1: 解析 JSON 文件名
│   └─ 得到 base_name = "IMG_20240913_162956(1).jpg", number = "1"
│
├─ 步骤2: 生成候选文件名列表
│   │
│   ├─ base_name 是否包含扩展名（如 .jpg）？
│   │   │
│   │   ├─ 是 → 分离 filename="IMG_20240913_162956", extension="jpg"
│   │   │   ├─ number 是否存在？
│   │   │   │   ├─ 是 → 候选1: "IMG_20240913_162956(1).jpg"
│   │   │   │   │   └─ 文件不存在 → 尝试修饰符变体:
│   │   │   │   │       ├─ "IMG_20240913_162956-已修改(1).jpg"
│   │   │   │   │       ├─ "IMG_20240913_162956-编辑(1).jpg"
│   │   │   │   │       ├─ "IMG_20240913_162956-修改(1).jpg"
│   │   │   │   │       ├─ "IMG_20240913_162956-edited(1).jpg"
│   │   │   │   │       └─ "IMG_20240913_162956-modified(1).jpg"
│   │   │   │   └─ 否 → 候选1: "IMG_20240913_162956.jpg"
│   │   │   │       └─ 文件不存在 → 尝试修饰符变体:
│   │   │   │           ├─ "IMG_20240913_162956-已修改.jpg"
│   │   │   │           ├─ "IMG_20240913_162956-编辑.jpg"
│   │   │   │           ├─ "IMG_20240913_162956-修改.jpg"
│   │   │   │           ├─ "IMG_20240913_162956-edited.jpg"
│   │   │   │           └─ "IMG_20240913_162956-modified.jpg"
│   │   │
│   │   └─ 否 → base_name 无扩展名
│   │       ├─ number 是否存在？
│   │       │   ├─ 是 → 候选1: "base_name(number)"
│   │       │   │   └─ 不存在 → 尝试5种修饰符变体
│   │       │   └─ 否 → 候选1: "base_name"
│   │       │       └─ 不存在 → 尝试5种修饰符变体
│   │
│   └─ 收集所有存在的候选文件 → photo_files 数组
│
├─ 步骤3: 验证候选结果
│   ├─ photo_files 数量 = 0 → 进入模糊匹配
│   ├─ photo_files 数量 = 1 → 采用该文件
│   └─ photo_files 数量 > 1 → 跳过（记录所有匹配文件路径，避免歧义误操作）
│
├─ 步骤4: 模糊匹配（仅当 photo_files 为空时执行）
│   │
│   ├─ 执行: find -maxdepth 1 -name "base_name*" ! -name "*.json"
│   │
│   ├─ 结果数量 = 1 → 直接采用（唯一匹配，安全）
│   │
│   ├─ 结果数量 = 0 且 number 存在 → 通配符匹配
│   │   ├─ 从 base_name 提取 prefix 和 suffix: base_name = "prefix(number)"
│   │   ├─ 执行: find -maxdepth 1 -name "prefix*(number)*" ! -name "*.json"
│   │   └─ 结果数量 = 1 → 采用
│   │
│   ├─ 结果数量 = 0 且 number 不存在 → 跳过（记录: "找不到对应的照片文件"）
│   │
│   └─ 结果数量 > 1 → 精确匹配筛选
│       ├─ number 存在？
│       │   ├─ 是 → 优先匹配:
│       │   │   1. candidate_name == base_name（完全相等）
│       │   │   2. candidate_name 以 base_name 开头 且 包含 "(number)"
│       │   └─ 否 → 优先匹配:
│       │       1. candidate_name == base_name（完全相等）
│       │       2. candidate_name 以 base_name 开头 且 不包含任何 "(数字)" 模式
│       │
│       └─ 筛选后结果数量 = 1 → 采用
│           └─ 筛选后结果数量 ≠ 1 → 跳过
│
└─ 输出: 最终匹配的照片文件路径，或标记为"未找到"
```

## 五、文件处理规则矩阵

### 5.1 完整规则表

| 文件类型 | 有配对 JSON | 时间戳来源优先级 | GPS/设备来源 | 无 JSON 时行为 |
|----------|-------------|------------------|--------------|----------------|
| IMG/VID 文件 | 是 | 1. 文件名（12种格式） → 2. JSON `.photoTakenTime.timestamp` | JSON `.geoData.*` + `.cameraMake` + `.cameraModel` | 从文件名提取时间戳，无 GPS/设备信息 |
| IMG/VID 文件 | 否 | 1. 文件名（12种格式） → 2. 失败则跳过 | 无 | 从文件名提取时间戳，无 GPS/设备信息 |
| 非 IMG/VID 文件 | 是 | 1. 文件名（12种格式） → 2. JSON `.photoTakenTime.timestamp` | JSON `.geoData.*` + `.cameraMake` + `.cameraModel` | 从文件名提取时间戳，无 GPS/设备信息 |
| 非 IMG/VID 文件 | 否 | 1. 文件名（12种格式） → 2. 失败则跳过 | 无 | 从文件名提取时间戳，无 GPS/设备信息 |

### 5.2 跳过条件清单

以下任一条件满足时，文件将被跳过：

| 序号 | 跳过条件 | 记录原因 | 计数器 |
|------|----------|----------|--------|
| 1 | JSON 解析后找不到对应的照片文件 | "找不到对应的照片文件 [base_name]" | skipped |
| 2 | JSON 解析后找到多个照片文件（歧义） | "找到多个匹配文件: [列表]" | skipped |
| 3 | JSON 中无 `.photoTakenTime.timestamp` 且文件名无法解析时间 | "无可用时间戳" | skipped |
| 4 | 文件名不匹配 12 种格式，且无 JSON 或 JSON 中无时间戳 | "无可用时间戳" | skipped |
| 5 | 提取的时间戳无效（如 2025-13-45） | "无效的时间戳: [值]" | error |
| 6 | 文件当前时间戳与目标时间戳差值在 [-1, +1] 秒内 | "时间戳已正确" | skipped |
| 7 | `touch -d` 执行失败 | "更新失败: [错误详情]" | error |
| 8 | 文件在扫描后、处理前被删除 | "文件不存在" | skipped |

## 六、命令行参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `--dry-run` | 布尔标志 | 关闭 | 预览模式：执行所有匹配和解析逻辑，但不调用 `touch` 修改文件时间戳。计数器正常累加。 |
| 无参数 | — | — | 正常模式：执行完整的时间戳修改流程。 |

### 6.1 dry-run 模式行为差异

| 操作 | 正常模式 | dry-run 模式 |
|------|----------|--------------|
| 文件扫描 | 执行 | 执行 |
| JSON 解析 | 执行 | 执行 |
| 照片文件匹配 | 执行 | 执行 |
| 时间戳提取 | 执行 | 执行 |
| 时间戳验证 | 执行 | 执行 |
| `touch -d` 调用 | 执行 | **不执行** |
| 成功计数 | processed++ | processed++ |
| 失败计数 | error++ | 不触发（不执行 touch） |
| 最终输出提示 | "成功修改了 N 个文件" | "将修改 N 个文件的时间戳" |

## 七、错误诊断体系

### 7.1 touch 失败时的诊断信息

当 `touch -d` 执行失败时，按顺序收集以下信息：

| 诊断项 | 命令 | 输出示例 | 用途 |
|--------|------|----------|------|
| 错误信息 | `touch -d "@timestamp" file 2>&1` | `touch: 无效日期` | 直接错误原因 |
| 文件权限 | `ls -la file` | `-rw-r--r-- 1 user user 12345` | 检查写权限 |
| 文件类型 | `file file` | `JPEG image data, JFIF standard 1.01` | 确认文件完整性 |
| 文件系统 | `df -T file \| tail -1 \| awk '{print $2}'` | `ntfs` | 检查文件系统兼容性 |

### 7.2 文件系统兼容性说明

| 文件系统类型 | 时间戳修改支持 | 注意事项 |
|--------------|----------------|----------|
| `ext4` | 完全支持 | Linux 原生文件系统，无兼容问题 |
| `ext3` | 完全支持 | 同上 |
| `xfs` | 完全支持 | 同上 |
| `btrfs` | 完全支持 | 同上 |
| `ntfs` | 部分支持 | WSL 挂载的 Windows NTFS 分区可能不支持纳秒级精度 |
| `vfat` | 部分支持 | FAT32/exFAT 文件系统，时间戳精度限制为 2 秒 |
| `9p` | 有限支持 | WSL 共享文件夹 (`/mnt/c/`) 使用的协议，某些操作可能失败 |

## 八、目录扫描范围

### 8.1 JSON 文件扫描

| 扫描起点 | 目录名称模式 | 递归深度 | 说明 |
|----------|--------------|----------|------|
| 当前工作目录 (`.`) | `Photos from*` | 仅一层（不递归进入子目录） | Google Takeout 导出的标准目录结构 |

**示例目录结构**：
```

├── Photos from 2023/          ← 扫描
│   ├── IMG_20230302_112040.jpg
│   ├── IMG_20230302_112040.json
│   └── SubFolder/             ← 不扫描（非 "Photos from*" 模式）
├── Photos from 2024/          ← 扫描
│   └── ...
└── OtherFolder/               ← 不扫描
```

### 8.2 IMG/VID 独立文件扫描

| 扫描起点 | 目录范围 | 递归深度 | 说明 |
|----------|----------|----------|------|
| 当前工作目录 (`.`) | 所有目录（包括 `Photos from*` 及其他） | 完全递归 | 确保覆盖无 JSON 配对的 IMG/VID 文件 |

### 8.3 去重机制

当同一文件同时出现在 JSON 队列和 IMG/VID 独立队列中时：

| 情况 | 处理方式 |
|------|----------|
| 文件已被 JSON 队列处理过 | IMG/VID 独立队列跳过该文件 |
| 文件仅在 IMG/VID 独立队列中 | 正常处理 |

**去重实现**：使用关联数组（`declare -A processed_files`）记录已处理文件的绝对路径。

## 九、时间戳精度与容差

### 9.1 时间精度

| 来源 | 精度 | 说明 |
|------|------|------|
| 文件名提取（2.1 格式1-2, IMG/VID 专用） | 秒级 | 从文件名中直接解析 `yyyyMMddHHmmss` 结构 |
| 文件名提取（2.1 格式3-4,6-8, 通用） | 秒级 | 从文件名中直接解析日期时间结构 |
| 文件名提取（2.1 格式5, WP 短时分秒） | 秒级 | 时间部分不足6位时，默认使用 12:00:00 |
| 文件名提取（2.1 格式9-11, mmexport） | 秒级 | 原始为毫秒级 Unix 时间戳，取前10位截断为秒级 |
| JSON `.photoTakenTime.timestamp` | 秒级 | Unix 时间戳（整数） |
| `stat -c %Y` 读取 | 秒级 | Linux mtime 精度 |
| `touch -d` 设置 | 秒级 | 设置 mtime 和 atime |

### 9.2 容差设置

| 场景 | 容差范围 | 原因 |
|------|----------|------|
| 判断是否需要更新 | ±1 秒 | 避免重复处理已正确设置的文件，容忍浮点舍入误差 |
| 日期有效性验证 | 无容差 | `date -d` 命令严格验证，无效日期（如 2月30日）直接拒绝 |

## 十、计数器与统计输出

### 10.1 计数器定义

| 计数器名称 | 触发条件 | 累加时机 |
|------------|----------|----------|
| `total_files` | 扫描阶段发现的待处理文件总数 | 扫描阶段 |
| `processed` | 时间戳成功应用（或 dry-run 下标记为可处理） | 处理阶段 |
| `skipped` | 因各种原因跳过（见 5.2 的条件 1-6, 8） | 处理阶段 |
| `error` | 时间戳有效但 `touch` 执行失败（见 5.2 的条件 5, 7） | 处理阶段 |

### 10.2 统计输出格式

```
🎉 处理完成!
📊 统计结果:
   总计: N 个文件
   ✅ 成功: N 个文件
   ⚠️  跳过: N 个文件
   ❌ 处理失败: N 个文件

[若 processed > 0]
✨ 成功修改了 N 个文件的时间戳
💡 提示: 可以使用 ls -la 命令查看文件的新时间戳

[若 dry-run 模式且 processed > 0]
✨ 将修改 N 个文件的时间戳
💡 提示: 重新运行脚本并去掉 --dry-run 参数以实际修改文件

[若 skipped > 0]
⚠️  有 N 个文件被跳过，可能的原因:
   • 找不到对应的照片文件
   • JSON 文件中没有时间戳信息
   • 文件名格式不匹配
   • 时间戳已经正确
   • 文件权限问题

[若 error > 0]
⚠️  有 N 个文件处理失败，可能的原因:
   • 文件权限问题
   • 无效的时间戳格式
   • 文件系统不支持时间戳修改
```

## 十一、临时文件管理

### 11.1 临时文件清单

| 临时文件路径 | 用途 | 创建时机 | 清理时机 |
|--------------|------|----------|----------|
| `/tmp/img_files_$$` | 存储 find 找到的所有 IMG/VID 文件列表（null 分隔） | 扫描阶段 | 处理完成后 |
| `/tmp/test_touch_$$` | 测试 touch -d 参数可用性 | 环境检查阶段 | 测试完成后立即清理 |

**说明**：`$$` 为当前进程 PID，确保多实例运行时不冲突。

### 11.2 异常清理

| 异常场景 | 清理策略 |
|----------|----------|
| 脚本正常结束 | 显式 `rm -f` 清理 |
| 脚本中途 `exit` | 在 exit 前显式 `rm -f` 清理 |
| 脚本被 `Ctrl+C` 中断 | 注册 `trap 'rm -f /tmp/img_files_$$' EXIT` |
| 脚本被 `kill -9` 终止 | 无法清理（僵尸文件，下次运行时因 PID 不同不会冲突） |

## 十二、边界条件处理

### 12.1 特殊文件名

| 文件名特征 | 处理方式 |
|------------|----------|
| 包含空格 | 使用 `find -print0` + `read -d ''` 处理（null 分隔） |
| 包含中文 | 正则 `[一-龥]` 匹配中文字符范围 |
| 包含括号 | 正则中使用 `\(` 和 `\)` 转义 |
| 包含多个扩展名 | `${basename%.*}` 仅去除最后一个扩展名 |
| 无扩展名 | `${basename%.*}` 返回原文件名，后续逻辑正常处理 |

### 12.2 极端场景

| 场景 | 处理方式 |
|------|----------|
| 当前目录无照片文件 | 输出 "未找到符合条件的文件" → exit 0 |
| JSON 文件存在但内容为空 | `jq` 返回空 → 跳过，记录 "JSON 解析失败" |
| JSON 文件存在但格式非法 | `jq` 报错 → 跳过，记录 "JSON 格式无效" |
| 照片文件为符号链接 | `find -type f` 不匹配符号链接 → 跳过 |
| 照片文件权限为只读 | `touch` 失败 → 记录错误，计数器 error++ |
| 同一目录存在同名不同扩展名的照片 | 精确匹配优先，模糊匹配阶段因多结果而跳过 |
| JSON 中的时间戳为 0 | `date -d "@0"` 返回 `1970-01-01 08:00:00`（UTC+8）→ 视为有效但异常 |
| 文件名中的时间为 24:00:00 | `date -d` 验证失败 → 跳过，记录 "无效的时间戳" |

## 十三、性能考量

### 13.1 时间复杂度

| 阶段 | 操作 | 时间复杂度 | 说明 |
|------|------|------------|------|
| 文件发现 | `find` 递归扫描 | O(N)，N 为文件总数 | 仅扫描，不读取内容 |
| JSON 解析 | 逐文件 `jq` 读取 | O(M)，M 为 JSON 文件数 | 每个 JSON 文件读取一次 |
| 照片匹配 | 精确匹配 O(1)，模糊匹配 O(K)，K 为同目录文件数 | O(M × K) | 最坏情况：每个 JSON 都触发模糊匹配 |
| 时间戳应用 | `touch` 系统调用 | O(P)，P 为待处理照片数 | 每次调用为 O(1) |

### 13.2 优化措施

| 优化点 | 措施 | 效果 |
|--------|------|------|
| 文件列表缓存 | `find` 结果写入临时文件，避免重复扫描 | 减少 50% 以上的 find 调用 |
| 精确匹配优先 | 先尝试精确文件名匹配，失败后才模糊匹配 | 90%+ 的场景下避免 O(K) 扫描 |
| 时间戳跳过 | 差值在 ±1 秒内不调用 touch | 减少重复运行时的系统调用 |
| null 分隔符 | `find -print0` + `read -d ''` | 正确处理含空格/换行符的文件名 |
