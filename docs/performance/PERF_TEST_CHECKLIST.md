# gtoh 性能优化 — 测试检查清单

## 一、方案 1：并发处理测试

### 1.1 单元测试

| 测试用例 | 输入 | 预期 |
|----------|------|------|
| 单文件 | 1 个文件 | 正确处理，无并发问题 |
| 多文件 | 10 个文件 | 所有文件处理完成，stats 正确 |
| 大量文件 | 100 个文件 | 无 race condition，stats 正确 |
| 空目录 | 0 个文件 | 正常退出 |

### 1.2 并发安全测试

```bash
# 使用 -race 检测
go test -race ./internal/migrator/
```

| 测试项 | 预期 |
|--------|------|
| stats 计数器 | 无 data race |
| logger 写入 | 无 data race，log 文件完整 |
| 进度条 | 无 data race，显示正常 |
| error 目录 | 并发创建不冲突 |
| metadata 目录 | 并发写入不冲突 |

### 1.3 性能测试

| 规模 | 串行耗时 | 并发耗时（4 workers） | 预期加速 |
|------|----------|----------------------|----------|
| 100 文件 | ~30s | ~10s | 3x |
| 1000 文件 | ~5min | ~1.5min | 3-4x |
| 10000 文件 | ~50min | ~12min | 4x |

## 二、方案 2：合并 exiftool 读取测试

### 2.1 单元测试

| 测试用例 | 输入 | 预期 |
|----------|------|------|
| 有时间戳+GPS | 包含 DateTimeOriginal 和 GPS 的 JPEG | 两者都正确解析 |
| 仅时间戳 | 仅包含 DateTimeOriginal 的 JPEG | 时间戳正确，GPS 为零 |
| 仅 GPS | 仅包含 GPS 的 JPEG | GPS 正确，时间戳为零 |
| 都无 | 无 EXIF 信息的文件 | 两者都返回零 |
| 解析失败 | 损坏的 JPEG | 返回错误 |

### 2.2 集成测试

| 测试项 | 预期 |
|--------|------|
| 合并调用结果与单独调用一致 | 时间戳和 GPS 值相同 |
| 回退机制 | 合并调用失败时单独调用仍能工作 |

## 三、方案 3：流式复制 + SHA-256 测试

### 3.1 单元测试

| 测试用例 | 输入 | 预期 |

| 1MB | ~10ms | ~5ms | 2x |
| 10MB | ~100ms | ~50ms | 2x |
| 100MB | ~1s | ~0.5s | 2x |
| 500MB | ~5s | ~2.5s | 2x |

## 五、方案 5：跳过不必要的 exiftool 读取测试

### 5.1 单元测试

| 测试用例 | 文件名 | 预期行为 |
|----------|--------|----------|
| IMG 格式 | `IMG_20240315_103000.jpg` | 不调用 exiftool，使用文件名时间戳 |
| VID 格式 | `VID_20240315_103000.mp4` | 不调用 exiftool，使用文件名时间戳 |
| Screenshot | `Screenshot_2024-03-15-10-30-00.png` | 不调用 exiftool，使用文件名时间戳 |
| 无时间戳 | `unknown.jpg` | 调用 exiftool 尝试读取 |
| 文件名无效 | `IMG_20241345_999999.jpg` | 文件名解析失败，调用 exiftool |

### 5.2 集成测试

| 测试项 | 预期 |
|--------|------|
| metadata 中 timestamp.source | 文件名可解析时为 "filename" |
| metadata 中 timestamp.exif | 文件名可解析时为空 |
| 优先级 | EXIF > Filename > JSON 不变 |
